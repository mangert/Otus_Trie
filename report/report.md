## Часть 2. Ассоциативный массив на базе префиксного дерева

### Архитектура

Проект реализует шаблонный класс `Trie` с поддержкой различных стратегий хранения детей в узлах:

#### Типы узлов

| Тип узла | Описание | Преимущества | Недостатки |
|----------|----------|--------------|------------|
| **`ArrayNode`** | Фиксированный массив на 26 указателей (a-z) | O(1) доступ, максимальная скорость | Память всегда 26 указателей, только латиница a-z |
| **`HashMapNode`** | `std::unordered_map<char, Node>` | Поддержка расширенного набора символов ключа, экономия памяти при разреженных данных | O(1) амортизировано, оверхед хеширования |
| **`HybridNode`** | Комбинация: массив для a-h + хеш-таблица для остальных | Баланс скорости и памяти для смешанных данных | Сложнее реализация |

#### Ключевые особенности реализации

1. **Семантика перемещения** — класс некопируемый, но перемещаемый
2. **Кастомные очистители** — возможность задать политику очистки значений (шаблонный параметр `Cleaner`)
3. **Итеративное удаление** — защита от переполнения стека при глубоких деревьях
4. **Поддержка некопируемых типов** — `std::unique_ptr` и другие через `find()`, возвращающий указатель
5. **Проверка глубины рекурсии** — защита от переполнения при клонировании

#### Ограничения на типы значений

- Тип значения должен быть **default constructible** (каждый узел содержит поле `value` даже без данных)
- Для некопируемых типов (`std::unique_ptr`) недоступны методы `search()` и `clone()`
- Для сырых указателей рекомендуется использовать кастомный очиститель или умные указатели

### Тестирование

Тестовый фреймворк `PrefixTest` проверяет:

- ✅ Корректность вставки/поиска/удаления для разных типов значений
- ✅ Работу с префиксами (`startsWith`)
- ✅ Семантику перемещения
- ✅ Кастомные очистители
- ✅ Клонирование (для копируемых типов)
- ✅ Очистку (`clear`)

#### Тестовые сценарии с кастомным очистителем

```cpp
// Логирующий очиститель для проверки
struct LoggingCleaner {
    void operator()(std::string& val) {
        std::cout << "  Cleaning: " << val << std::endl;
    }
};
```

Проверены три сценария:
1. **Очистка старого значения при перезаписи ключа**
2. **Очистка при удалении ключа**
3. **Очистка всех значений при разрушении дерева**

### Сравнительный анализ производительности

Тестирование проводилось на 2000 английских словах (из файла `words.txt`) в **Release-конфигурации**.  
Сравнивались три реализации префиксного дерева с хеш-таблицей `OpenHashTable`.

#### Время выполнения (мкс)

| Операция | ArrayNode | HashMapNode | HybridNode | OpenHashTable |
|----------|-----------|-------------|------------|---------------|
| **Insert (2000 ключей)** | 2249 | 3488 | 3598 | **888** |
| **Find (10% ключей)** | 18 | **0*** | **0*** | 28 |
| **Remove (10% ключей)** | 53 | 172 | 69 | **19** |

\* *Нулевое время — артефакт измерения (менее 1 мкс), фактически поиск в HashMapNode и HybridNode выполняется за время, сравнимое с ArrayNode*

#### Наблюдения

1. **ArrayNode** — ожидаемо быстро для латиницы (прямая индексация)
2. **HashMapNode** — медленнее на вставке и удалении из-за хеширования, но поиск сопоставим
3. **HybridNode** — промежуточный результат, близкий к HashMapNode
4. **OpenHashTable** — быстрее всех на всех операциях, но **не поддерживает** `startsWith`

### Выводы

- Префиксное дерево — эффективная структура для задач с **поиском по префиксу**
- Выбор типа узла зависит от алфавита:
  - Фиксированный латинский алфавит → `ArrayNode`
  - Произвольные символы → `HashMapNode`
  - Смешанный алфавит с частотными первыми буквами → `HybridNode`
- Для максимальной производительности поиска хеш-таблица предпочтительнее, но она не решает задачу автодополнения
- Реализация с кастомными очистителями делает класс гибким для разных сценариев управления памятью
